## Arc-Enabled Kubernetes

Azure Arc-enabled Kubernetes lets you attach and manage **any CNCF-conformant Kubernetes cluster** — running on-premises, at the edge, or in other clouds — through Azure Resource Manager. Once connected, clusters can leverage Azure services like GitOps configuration management, Azure Policy for Kubernetes, cluster extensions, and centralized monitoring.

## Supported Kubernetes Distributions

Azure Arc supports any CNCF-conformant cluster, including:

- **Managed cloud**: AKS on Azure Stack HCI, EKS (AWS), GKE (Google Cloud)
- **Self-managed**: kubeadm, Rancher/RKE2, OpenShift, Tanzu, MicroK8s
- **Lightweight/edge**: K3s, K0s, MicroK8s, Kind (development only)

The cluster must run Kubernetes **1.26 or later** (check the Arc docs for current supported versions).

## Connecting a Cluster

### Prerequisites

- Azure CLI with the `connectedk8s` extension
- `kubectl` configured with cluster-admin access to the target cluster
- Outbound connectivity from the cluster to Azure on HTTPS (TCP 443)
- `Microsoft.Kubernetes` and `Microsoft.KubernetesConfiguration` resource providers registered

### Connect the Cluster

```bash
# Install the CLI extension
az extension add --name connectedk8s

# Register resource providers (one-time per subscription)
az provider register --namespace Microsoft.Kubernetes
az provider register --namespace Microsoft.KubernetesConfiguration

# Connect the cluster
az connectedk8s connect \
  --name "prod-cluster-seattle" \
  --resource-group "rg-arc-k8s" \
  --location "westus2" \
  --tags "Environment=Production Region=Seattle"
```

This deploys Arc agents into the `azure-arc` namespace on the cluster. The agents maintain an outbound connection to Azure and sync resource metadata.

```bash
# Verify the connection
az connectedk8s show \
  --name "prod-cluster-seattle" \
  --resource-group "rg-arc-k8s" -o table

# Check agent pods
kubectl get pods -n azure-arc
```

## Cluster Extensions

Cluster extensions are the mechanism for deploying Azure service integrations onto Arc-enabled Kubernetes clusters. They use Helm charts under the hood but are managed through ARM.

```bash
# Install Azure Monitor Container Insights
az k8s-extension create \
  --name "azuremonitor-containers" \
  --cluster-name "prod-cluster-seattle" \
  --resource-group "rg-arc-k8s" \
  --cluster-type connectedClusters \
  --extension-type Microsoft.AzureMonitor.Containers \
  --configuration-settings "logAnalyticsWorkspaceResourceID=/subscriptions/SUB/resourceGroups/RG/providers/Microsoft.OperationalInsights/workspaces/WS"

# Install Azure Key Vault Secrets Provider
az k8s-extension create \
  --name "akvsecretsprovider" \
  --cluster-name "prod-cluster-seattle" \
  --resource-group "rg-arc-k8s" \
  --cluster-type connectedClusters \
  --extension-type Microsoft.AzureKeyVaultSecretsProvider
```

### Available Cluster Extensions

| Extension | Purpose |
|---|---|
| Azure Monitor Container Insights | Cluster and container-level monitoring and logs |
| Microsoft Defender for Containers | Runtime threat protection, vulnerability scanning |
| Azure Policy for Kubernetes | OPA Gatekeeper-based policy enforcement |
| Azure Key Vault Secrets Provider | Mount Key Vault secrets as Kubernetes secrets |
| Azure Arc data services | SQL MI and PostgreSQL on Kubernetes |
| Flux (GitOps) | Declarative configuration from Git repositories |
| Open Service Mesh | Lightweight SMI-compatible service mesh |
| Dapr | Distributed application runtime for microservices |

## GitOps with Flux v2

GitOps is the recommended approach for deploying workloads and configurations to Arc-enabled Kubernetes clusters. Azure Arc uses **Flux v2** as the GitOps engine.

```bash
# Create a Flux configuration pointing to a Git repo
az k8s-configuration flux create \
  --name "cluster-config" \
  --cluster-name "prod-cluster-seattle" \
  --resource-group "rg-arc-k8s" \
  --cluster-type connectedClusters \
  --namespace "flux-system" \
  --scope cluster \
  --url "https://github.com/myorg/k8s-configs" \
  --branch "main" \
  --kustomization name=infra path=./infrastructure prune=true \
  --kustomization name=apps path=./apps prune=true dependsOn=infra
```

See the dedicated [GitOps page](/azure-arc-overview/gitops) for a deeper dive into Flux v2 configuration.

## Azure Policy for Kubernetes

Apply Azure Policy to Arc-enabled clusters to enforce guardrails:

- **Require resource limits** on all containers
- **Deny privileged containers** across namespaces
- **Enforce allowed container registries** (e.g., only from your ACR)
- **Require specific labels** on namespaces and pods
- **Audit/deny workloads** without network policies

Azure Policy for Kubernetes uses **OPA Gatekeeper** under the hood, deployed automatically when you assign your first Kubernetes policy.

```bash
# Enable Azure Policy extension on the cluster
az k8s-extension create \
  --name "azure-policy" \
  --cluster-name "prod-cluster-seattle" \
  --resource-group "rg-arc-k8s" \
  --cluster-type connectedClusters \
  --extension-type Microsoft.PolicyInsights
```

## Cluster Connect (Reverse Proxy)

**Cluster Connect** enables you to access the Kubernetes API server of an Arc-enabled cluster without exposing it to the internet. It creates a reverse proxy tunnel through the Arc agents.

```bash
# Enable cluster connect
az connectedk8s enable-features \
  --name "prod-cluster-seattle" \
  --resource-group "rg-arc-k8s" \
  --features cluster-connect

# Get a kubeconfig that routes through Azure
az connectedk8s proxy \
  --name "prod-cluster-seattle" \
  --resource-group "rg-arc-k8s" \
  --token "YOUR_SERVICE_ACCOUNT_TOKEN"
```

This is especially valuable for clusters behind firewalls or in restricted networks — you can use the Azure portal's Kubernetes resource viewer, run Helm commands, or use `kubectl` remotely via the Arc proxy.

## Custom Locations

**Custom locations** map an Arc-enabled Kubernetes cluster (or a specific namespace) to an Azure location. This enables deployment of Azure PaaS services like App Service, Functions, Logic Apps, API Management, and Arc data services onto your own infrastructure.

```bash
# Create a custom location
az customlocation create \
  --name "seattle-dc-location" \
  --resource-group "rg-arc-k8s" \
  --namespace "arc-services" \
  --host-resource-id "/subscriptions/SUB/resourceGroups/rg-arc-k8s/providers/Microsoft.Kubernetes/connectedClusters/prod-cluster-seattle" \
  --cluster-extension-ids "/subscriptions/SUB/resourceGroups/rg-arc-k8s/providers/Microsoft.Kubernetes/connectedClusters/prod-cluster-seattle/providers/Microsoft.KubernetesConfiguration/extensions/arc-data-services"
```

## Best Practices

- **Use GitOps** for all workload deployments — avoid `kubectl apply` in production
- **Assign Azure Policy** initiatives to enforce security baselines on all clusters
- **Enable Defender for Containers** for runtime threat detection
- **Label and tag clusters** consistently for cost tracking and resource organization
- **Monitor agent health** via Azure Monitor and set up alerts for disconnected clusters
- **Use RBAC** with Azure AD authentication for cluster access via Cluster Connect
