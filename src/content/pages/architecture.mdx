## Network Connectivity Options

Azure Arc agents communicate outbound-only over HTTPS (port 443). There are four connectivity options depending on your network requirements:

![Azure Arc-enabled server network connectivity options](/azure-arc-overview/images/arc-network-connectivity.png)

*Source: [Arc-enabled servers connectivity — Cloud Adoption Framework (Microsoft Learn)](https://learn.microsoft.com/en-us/azure/cloud-adoption-framework/scenarios/hybrid/arc-enabled-servers/eslz-arc-servers-connectivity#architecture)*

| Option | Path | When to Use |
|--------|------|-------------|
| **① Direct** | Server → Internet → Public endpoint | Simple environments, no proxy/firewall restrictions |
| **② Proxy** | Server → Proxy → Internet → Public endpoint | Environments requiring traffic inspection or URL filtering |
| **③ Service Tag** | Server → S2S VPN/ExpressRoute → Azure VNet → Public endpoint | Enterprises with private connectivity to Azure, using NSG service tags |
| **④ Private Link** | Server → S2S VPN/ExpressRoute → Azure VNet → Private endpoint | Most secure — all Arc traffic stays on private network |

## Azure Landing Zone Integration

Azure Arc fits into the **Azure Landing Zone** conceptual architecture:

- **Platform Landing Zone** — Hosts shared services: Log Analytics workspace, Automation account, Private DNS zones, Private Link endpoints
- **Application Landing Zones** — Dedicated subscriptions for Arc-enabled resource groups, organized by environment or business unit
- **Management Group hierarchy** — Policies assigned at the top-level management group inherit to all Arc subscriptions

```
Root MG
├── Platform
│   ├── Management (Log Analytics, Automation, Defender)
│   ├── Connectivity (Private Link, DNS, Firewall)
│   └── Identity (Azure AD, PIM)
├── Arc Landing Zones
│   ├── Arc-Servers-Prod
│   ├── Arc-Servers-NonProd
│   ├── Arc-SQL-Prod
│   └── Arc-Data-Prod
└── Sandbox
```

## Azure Private Link

**Azure Arc Private Link Scope** enables Arc agents to communicate with Azure entirely over private network paths, using Azure ExpressRoute or VPN connections.

### Setting Up Private Link

```bash
# Create a Private Link Scope
az connectedmachine private-link-scope create \
  --name "pls-arc-production" \
  --resource-group "rg-arc-networking" \
  --location "eastus"

# Create a Private Endpoint in your hub VNet
az network private-endpoint create \
  --name "pe-arc-hub" \
  --resource-group "rg-arc-networking" \
  --vnet-name "vnet-hub" \
  --subnet "snet-private-endpoints" \
  --private-connection-resource-id "/subscriptions/SUB/resourceGroups/rg-arc-networking/providers/Microsoft.HybridCompute/privateLinkScopes/pls-arc-production" \
  --group-id "hybridcompute" \
  --connection-name "arc-pls-connection"

# Associate Arc-enabled servers with the Private Link Scope
az connectedmachine private-link-scope update-tag \
  --name "pls-arc-production" \
  --resource-group "rg-arc-networking"
```

### Private DNS Zones

Private Link for Arc requires these private DNS zones:

- `privatelink.his.arc.azure.com`
- `privatelink.guestconfiguration.azure.com`
- `privatelink.dp.kubernetesconfiguration.azure.com`

These DNS zones must be linked to the VNet where your Private Endpoint resides, and on-premises DNS must forward queries for these domains to Azure DNS (168.63.129.16) via a DNS forwarder in the hub VNet.

## Azure Arc Gateway

For enterprises that use proxies to manage outbound traffic, **Azure Arc gateway** simplifies network configuration by reducing the required endpoints from dozens to just **seven FQDNs**. Instead of opening firewall rules for every Arc service endpoint, you create a single gateway resource that serves as a common front-end.

### How It Works

Traffic flows through the gateway chain:

**Arc agents → Arc proxy (local) → Enterprise proxy → Arc gateway → Target service**

![Azure Arc gateway traffic flow](/azure-arc-overview/images/arc-gateway-overview.png)

*Source: [Azure Arc gateway (Microsoft Learn)](https://learn.microsoft.com/en-us/azure/azure-arc/servers/arc-gateway)*

The gateway has two components:
- **Arc gateway resource** — An Azure resource that serves as the common front-end. Created in your subscription and served on a specific domain (`<prefix>.gw.arc.azure.com`)
- **Arc proxy** — A local service added to the Arc agent that acts as a forward proxy. No configuration needed on your part

### Required Endpoints (with Gateway)

With Arc gateway enabled, you only need to allow these **7 endpoints** through your firewall:

| Endpoint | Purpose |
|----------|---------|
| `<PREFIX>.gw.arc.azure.com` | Your Arc gateway endpoint |
| `management.azure.com` | Azure Resource Manager |
| `login.microsoftonline.com` | Microsoft Entra ID authentication |
| `gbl.his.arc.azure.com` | Global service discovery |
| `packages.microsoft.com` | Linux agent packages |
| `mcr.microsoft.com` | Container images |
| `*.blob.core.windows.net` | Extension downloads |

Compare this to the full endpoint list (without gateway) that requires opening access to a dozen or more wildcard domains.

### Creating an Arc Gateway Resource

```bash
# Create the gateway resource
az connectedmachine gateway create \
  --name "arc-gateway-prod" \
  --resource-group "rg-arc-networking" \
  --location "eastus" \
  --gateway-type "Public" \
  --allowed-features "*"
```

The response includes the gateway URL (`<prefix>.gw.arc.azure.com`). Use this when onboarding servers:

```bash
# Onboard a server using the gateway
azcmagent connect \
  --resource-group "rg-arc-servers" \
  --location "eastus" \
  --gateway-id "/subscriptions/{sub-id}/resourceGroups/rg-arc-networking/providers/Microsoft.HybridCompute/gateways/arc-gateway-prod"
```

For existing Arc-enabled servers, associate them with the gateway:

```bash
# Associate an existing server with the gateway
azcmagent config set connection.gateway "/subscriptions/{sub-id}/resourceGroups/rg-arc-networking/providers/Microsoft.HybridCompute/gateways/arc-gateway-prod"
```

### Limitations

- **No proxy bypass** — When Arc gateway is in use, `azcmagent config set proxy.bypass` has no effect
- **5 gateway resources per subscription** — Plan gateway placement across subscriptions
- **Public cloud only** — Arc gateway is available in Azure public cloud, not sovereign clouds
- **TLS inspection** — Not recommended with Arc gateway. If required, skip TLS inspection for `<prefix>.gw.arc.azure.com`
- Some extensions may still require additional endpoints beyond the core 7

## Network Security Best Practices

- **Use Private Link** for production environments with ExpressRoute or VPN connectivity
- **Centralize DNS** in the hub VNet for Private Link DNS resolution
- Place Arc-enabled servers in **dedicated subnets** with NSG rules
- Combine Azure Firewall with Private Link for **TLS inspection and logging** (exclude certificate-pinned endpoints)
- **Document firewall rules** and test connectivity before large-scale agent deployment
- **Select the nearest Azure region** for Arc resource registration to minimize latency
- **Monitor network connectivity** — Use `azcmagent check` to validate endpoint reachability before onboarding

## Agent Resilience

- **Server agent** — Caches state locally and reconnects automatically after network interruptions. Reports as "Disconnected" in the portal after 5 minutes without heartbeat
- **Kubernetes agents** — Run as Kubernetes deployments with replica counts. Tolerate brief cluster network interruptions
- **Multi-region** — Register Arc resources in the Azure region closest to your datacenter for lowest latency
